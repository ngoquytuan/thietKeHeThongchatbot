## 3. METADATA SCHEMA DESIGN

### 3.1 Core Metadata Structure

#### 3.1.1 Primary Metadata Table
```sql
-- Table: documents_metadata
CREATE TABLE documents_metadata (
    -- Primary identifier
    document_id VARCHAR(50) PRIMARY KEY,
    
    -- Source information
    source_file VARCHAR(500) NOT NULL,
    original_filename VARCHAR(255) NOT NULL,
    file_hash SHA256 UNIQUE,
    
    -- Version control
    version VARCHAR(20) NOT NULL DEFAULT '1.0',
    version_sequence INTEGER NOT NULL DEFAULT 1,
    
    -- Ownership & governance
    department_owner VARCHAR(100) NOT NULL,
    author VARCHAR(100) NOT NULL,
    reviewer VARCHAR(100),
    approver VARCHAR(100),
    
    -- Access control
    access_level ENUM('public', 'employee_only', 'manager_only', 'director_only') NOT NULL,
    
    -- Classification
    document_type ENUM('policy', 'procedure', 'technical_guide', 'report') NOT NULL,
    category VARCHAR(100),
    subcategory VARCHAR(100),
    
    -- Content metadata
    title VARCHAR(500) NOT NULL,
    description TEXT,
    language CHAR(2) DEFAULT 'vi',
    
    -- Timestamps
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    last_updated TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    published_at TIMESTAMP,
    expires_at TIMESTAMP,
    
    -- Status management
    status ENUM('draft', 'review', 'approved', 'published', 'archived') DEFAULT 'draft',
    
    -- Search optimization
    search_vector TSVECTOR,
    
    -- Flexible attributes (JSON)
    custom_attributes JSONB,
    
    -- Audit
    created_by VARCHAR(100) NOT NULL,
    updated_by VARCHAR(100) NOT NULL
);
```

#### 3.1.2 Tags System (Many-to-Many Relationship)
```sql
-- Table: tags
CREATE TABLE tags (
    tag_id SERIAL PRIMARY KEY,
    tag_name VARCHAR(50) UNIQUE NOT NULL,
    tag_category VARCHAR(50),
    description TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Table: document_tags (junction table)
CREATE TABLE document_tags (
    document_id VARCHAR(50) REFERENCES documents_metadata(document_id) ON DELETE CASCADE,
    tag_id INTEGER REFERENCES tags(tag_id) ON DELETE CASCADE,
    added_by VARCHAR(100) NOT NULL,
    added_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (document_id, tag_id)
);
```

#### 3.1.3 Version History Tracking
```sql
-- Table: document_versions
CREATE TABLE document_versions (
    version_id SERIAL PRIMARY KEY,
    document_id VARCHAR(50) REFERENCES documents_metadata(document_id),
    version VARCHAR(20) NOT NULL,
    version_sequence INTEGER NOT NULL,
    change_summary TEXT,
    changed_by VARCHAR(100) NOT NULL,
    changed_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    previous_metadata JSONB -- Snapshot of previous metadata
);
```

### 3.2 Supporting Tables

#### 3.2.1 Department Management
```sql
-- Table: departments
CREATE TABLE departments (
    department_id VARCHAR(50) PRIMARY KEY,
    department_name VARCHAR(100) NOT NULL,
    parent_department_id VARCHAR(50) REFERENCES departments(department_id),
    department_head VARCHAR(100),
    contact_email VARCHAR(100),
    is_active BOOLEAN DEFAULT true
);
```

#### 3.2.2 Access Control Matrix
```sql
-- Table: access_permissions
CREATE TABLE access_permissions (
    permission_id SERIAL PRIMARY KEY,
    user_role ENUM('guest', 'employee', 'manager', 'director', 'admin'),
    access_level ENUM('public', 'employee_only', 'manager_only', 'director_only'),
    can_read BOOLEAN DEFAULT false,
    can_write BOOLEAN DEFAULT false,
    can_approve BOOLEAN DEFAULT false
);
```

## 4. BUSINESS RULES VÀ VALIDATION

### 4.1 Core Business Rules

#### 4.1.1 Document ID Generation Rules
```yaml
Document ID Format: {department_code}_{doc_type}_{sequence}_{year}
Examples:
  - RD_PROC_001_2025    # R&D Procedure
  - HR_POLICY_015_2025  # HR Policy
  - IT_GUIDE_008_2025   # IT Technical Guide
  
Rules:
  - Must be unique across entire system
  - Cannot be changed after creation
  - Must follow naming convention
  - Maximum 50 characters
```

#### 4.1.2 Version Control Rules
```yaml
Version Format: {major}.{minor}[.{patch}]
Examples:
  - "1.0"     # Initial version
  - "1.1"     # Minor update
  - "2.0"     # Major revision
  - "1.0.1"   # Patch/hotfix

Rules:
  - Version must increment logically
  - Major version change requires approval
  - Cannot skip version numbers
  - Previous versions remain accessible
```

#### 4.1.3 Access Level Hierarchy
```yaml
Access Hierarchy (top can access lower levels):
  director_only:
    - Can access: director_only, manager_only, employee_only, public
    - Users: Directors, C-level executives
  
  manager_only:
    - Can access: manager_only, employee_only, public
    - Users: Department heads, Team leaders
  
  employee_only:
    - Can access: employee_only, public
    - Users: Regular employees
  
  public:
    - Can access: public only
    - Users: Guests, external users, unauthenticated
```
